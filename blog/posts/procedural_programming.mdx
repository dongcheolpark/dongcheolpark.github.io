## 들어가며

혹시 절차적 프로그래밍을 '주어진 명령을 순서대로 수행하는 프로그래밍 패러다임' 이라고 생각하고 있지는 않은가? 잘 고민해보자. 프로그래밍에서 '순서'라는 개념을 제외할 수 있을까?

또 객체지향 프로그래밍을 배우기 이전에 행하던 프로그래밍 방식을 절차적 프로그래밍이라고 이해하는 경우가 있다. 그러나 객체지향 프로그래밍와 절차적 프로그래밍은 상당히 밀접한 관계가 있으며 절차적 프로그래밍에 대해 잘 이해하면 객체지향 프로그래밍을 이해하기도 쉬워진다. 한 번 알아보자.

두 경우는 절차적 프로그래밍과 관련된 대표적인 오해이다. 이 글을 읽고나면 위 오해를 깔끔하게 해결할 수 있을 것이다.

## 절차적 프로그래밍이란

`절차적 프로그래밍`에 대해서 먼저 정의를 해보자. 절차적 프로그래밍은 영어로 `procedural programming` 이라고 한다. 여기서 `procedure`, 프로시저라 함은 어떤 곳에서 호출할 수 있는 독립된 코드 조각이라고 할 수 있다.

따라서 절차적 프로그래밍은 프로시저를 사용해서 프로그래밍하는 방법론이라고 할 수 있겠다.

> 혹자는 `절차지향 프로그래밍`은 잘못된 표현이라고 하는데, 나는 '절차적', '절차지향'의 의미가 큰 차이가 있는지 잘 모르겠다.

프로시저가 어떤 장점이 있기에 절차적 프로그래밍이 탄생한 것일까?

## 태초에 순차적 프로그래밍이 있었다.

순차적 프로그래밍에서는 하나의 statement가 실행되고 또 그다음 statement가 실행되고... 이를 반복한다. 이로 인한 단점이 크게 두 가지가 생긴다.

1. 반복적으로 쓰이는 코드 덩어리가 생긴다.

2. 프로그램의 크기가 커질수록 어떤 코드가 무슨 일을 하는지 구분하기 어렵다.

이를 해결하기 위해 과거의 프로그래머들은 프로시저라는 개념을 탄생시켰다.

프로시저를 바구니라고 생각해보자. 코드를 프로시저로 묶는다는건 흩여져있는 물건들을 잘 정리해서 바구니안에 담은것이다.

바구니에 잘 담아서 바구니마다 이름표를 붙여두니 어떤 물건이 어딨는지 찾고 싶을 때 찾기 쉬워졌다.

프로시저는 바구니보다 더 유용한 존재이다. 프로시저는 복사가 된다. 같은 일을 여러 군데에서 하고 있다면 복사 - 붙여넣기 대신 프로시저를 호출하면 된다.

### 함수

요즘에는 프로시저라는 말을 잘 쓰지 않는다. 대신 서브 루틴 그리고 함수라는 용어를 많이 사용한다. 이 둘은 프로시저의 한 종류라고 할 수 있다.

둘의 의미는 프로그래머들 사이에도 통일되지 못했고 사람마다 의견이 다르다. 다만 나는 서브 루틴을 '명령 조각들을 실행하는 모듈'이라고 정의하고 함수는 '입력에 따라 결과를 계산하는 모듈' 이라고 정의한다.

아래 간단한 예시를 보자.

```js
// shop.js

function printMenu() {
  console.log('---메뉴판---');
  console.log('메뉴를 선택해 주세요.');
  console.log('순대 국밥');
  console.log('돼지 국밥');
  console.log('뼈해장국');
}

function calculatePrice(menuPrice, numberOfCustomer) {
  return menuPrice * numberOfCustomer;
}
```

js에서는 서브루틴이라는 표현은 쓰지 않고 전부 함수라고 표현한다. 위 두 함수는 모두 함수다. 그러나 둘이 하는 일은 사뭇 다르다.

`printMenu` 함수는 console.log를 사용해서 콘솔에 문자열을 출력하고 있다. 리턴값도 없다.
